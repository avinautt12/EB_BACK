 🔧 FUNCIONES NUEVAS - DESCRIPCIÓN DETALLADA
═══════════════════════════════════════════════════════════════════════════════

1. obtener_usuarios_monitor()
   ─────────────────────────────
   Ubicación: utils/otp_utils.py líneas 95-150
   
   Propósito:
     Obtiene TODOS los usuarios visibles en Monitor de Pedidos.
     Incluye: Clientes con usuario, clientes sin usuario, usuarios integrales.
   
   Entrada: Ninguna
   Salida: list[dict] con estructura:
     [{
       "id_usuario": 1,
       "nombre": "Juan Pérez",
       "usuario": "juan.perez",
       "activo": true,
       "clave": "CLI-001",
       "id_grupo": 1,
       "nombre_grupo": "Grupo A",
       "rol_id": 2
     }, ...]
   
   Diferencia fundamental:
     • obtener_usuarios_monitor() → Obtiene SIN FILTRO
     • listar_tokens_usuarios() → Obtiene CON FILTRO rol_id != 1
     Por eso necesitamos esta nueva función: incluye todos.


2. sincronizar_usuarios_desde_monitor()
   ────────────────────────────────────
   Ubicación: utils/otp_utils.py líneas 153-188
   
   Propósito:
     Pre-genera OTP para TODOS los usuarios del Monitor.
     Invalida OTPs previos automáticamente.
   
   Entrada: Ninguna
   Salida: dict con estructura:
     {
       "sincronizados": 45,
       "errores": 0,
       "total_monitor": 45,
       "detalles": [{
         "id_usuario": 1,
         "nombre": "Juan Pérez",
         "estado": "sincronizado",
         "codigo": "654321"
       }, ...]
     }
   
   Lógica:
     1. Obtiene usuarios del Monitor
     2. Para cada usuario: generar_otp(usuario_id)
     3. Captura errores si existen
     4. Retorna reporte con detalles


3. listar_tokens_usuarios_monitor()
   ────────────────────────────────
   Ubicación: utils/otp_utils.py líneas 191-238
   
   Propósito:
     Retorna usuarios del Monitor CON sus OTPs vigentes.
     Diferente de listar_tokens_usuarios() que filtra por rol_id.
   
   Entrada: Ninguna
   Salida: list[dict] con estructura:
     [{
       "id": 1,
       "nombre": "Juan Pérez",
       "usuario": "juan.perez",
       "activo": true,
       "clave": "CLI-001",
       "nombre_grupo": "Grupo A",
       "token_activo": "654321",
       "expira_en": "2025-01-20 14:30:00",
       "creado_en": "2025-01-20 13:30:00"
     }, ...]
   
   Diferencia clave:
     • listar_tokens_usuarios() → rol_id != 1 (restrictivo)
     • listar_tokens_usuarios_monitor() → TODOS (completo)

 🌐 ENDPOINTS NUEVOS - API REFERENCE
═══════════════════════════════════════════════════════════════════════════════

1. POST /edicion/sincronizar-desde-monitor
   ーーーーーーーーーーーーーーーーーーーーーーー

   Descripción:

     • Cuando reportan "faltan usuarios en edición"

       },

     ]
   
   Casos de uso:
     • Mostrar dashboard de editores
     • Actualizar UI con usuarios sincronizados
     • Monitorear OTPs vigentes en tiempo real
